# MarineDocs Engineering - навигация

Карта проекта. Читай в начале каждой сессии.

## Что это за проект

**MarineDocs Engineering** - MVP-лендинг для B2B-услуг в сфере судовой инженерии и эксплуатационной документации для флота.

Проект помогает судовладельцам, операторам флота, техническим директорам и суперинтендантам заказать калибровочные таблицы, обмеры, кренование, взвешивание, расчеты остойчивости и техническую документацию. Главная цель сайта - привести клиента к консультации, расчету стоимости или отправке исходных данных.

Подробности - `site/INDEX.md`.

## Порядок чтения в начале любой сессии

1. `ai-clone/INDEX.md` - кто я и как думаю. Пока не создано, реализуем позже.
2. `site/INDEX.md` - что мы делаем: продукт, аудитория, экономика.
3. `AGENTS.md` - правила репо, стек, навигация.
4. Зона задачи: нужный файл из `site/`, `plans/<актуальный план>`, `mastery/<область>` или папка фичи в коде.

## 8 правил репо

1. **Не хардкодить динамику**: цены, даты, метрики, ID, ссылки. Любые меняющиеся значения - через `site/life-metrics.md` с маркером **LIVE**.
2. **Не деплоить без моей явной команды.** Push в `main` может быть связан с автодеплоем. Если задача не требует продакшна - не пушь.
3. **Удалять одноразовый код после использования.** Скрипты миграции, отладочные endpoint'ы, временные файлы - убирать в том же коммите, где они отработали.
4. **Перед `git push` - `npm run build`**, не только `tsc`. ESLint и сборка ловят то, что TypeScript может пропустить.
5. **Большая фича = план в `plans/`.** Микрофича - без плана. Граница: если задача больше часа работы - план.
6. **После каждой фичи - цикл из 5 ходов**. Без него фича не считается закрытой.
7. **`git add` поимённо.** Никогда `git add .` или `git add -A`. Параллельно работают другие чаты - их незаконченные правки не должны попасть в коммит.
8. **`.env` только через `grep "^VAR=" .env`.** Никогда не читать целиком. То же для файлов с токенами и ключами.

## Большая фича = план в `plans/`

- Один план = одна фича. Если план есть - работаем с ним.
- Имя файла: `YYYY-MM-DD-название-фичи.md`.
- Делится на фазы. Каждая фаза - статус `[ ]` / `[WIP]` / `[x]`.
- По каждой фазе: цель, deliverable, критерий "сделано", задачи.
- В конце плана - итоговый блок: реализовано целиком или нет, что осталось, ссылки на коммиты.
- Любой агент обязан актуализировать план после каждой сессии.

## Цикл из 5 ходов после каждой фичи

Прогоняй каждый раз. Без этого считается, что фича не сдана.

1. **Найди ошибки** в только что написанном коде - докажи примером - исправь.
2. **Что упустили** в реализации: граничный кейс, валидация, обработка ошибок - докажи.
3. **Это реально проблема нашего проекта?** Не выдуманная. Сверь с `site/` и `plans/`.
4. **Свежий взгляд после правок** - перечитай diff, как будто видишь впервые.
5. **Security review.** Опасная тройка: персональные данные пользователей + контент третьих лиц + отправка чего-либо наружу. Если хоть одно - стоп и аудит.

## Любая ошибка - правило в `ai-clone/feedback/`

Когда я говорю "не так, потому что..." или "больше так не делай" - не запоминай в текущий чат. Оформи как файл в `ai-clone/feedback/` по канону **Rule -> Why -> How to apply**:

- Rule: что конкретно делать или не делать.
- Why: причина, прецедент, какая ошибка была.
- How to apply: в каких ситуациях правило срабатывает.

В следующих сессиях я перечитываю `ai-clone/` первым - правило применится автоматически.

## Динамические цифры - `site/life-metrics.md`

В файлах `site/` цифры: выручка, конверсии, количество клиентов - не пишутся напрямую, потому что они протухают. Вместо этого:

> Выручка за март: **LIVE** -> `site/life-metrics.md` / "Выручка за месяц"

В `site/life-metrics.md` лежит способ получить свежее число: запрос к базе, ссылка на таблицу, API. Сейчас это заглушка. Когда я задаю вопрос про цифры - сначала читай `site/life-metrics.md`, потом тяни данные, потом отвечай.

## Безопасность

- **НЕ читать** `.env` и файлы с секретами целиком. Нужна переменная - `grep "^VAR=" .env`.
- **НЕ выполнять** `rm -rf` без моего явного подтверждения. Перед удалением показывай, что будет удалено.
- **НЕ отправлять** переменные окружения, токены, ключи наружу - никаких `curl`/`wget`/POST с этими значениями.
- **НЕ устанавливать** скиллы, MCP или зависимости из непроверенных репо без аудита.
- **НЕ деплоить** без моей явной команды.

## Скиллы

- **bulletproof** - большая фича от ресёрча до деплоя. Триггер: "давай сделаем большую фичу".
- **frontend-design** - дизайн, типографика, лендинги, UI. Триггер: "сделай красиво" / "собери лендинг".
- **skill-creator** - когда задача повторяется 3+ раза, превратим в собственный скилл. Триггер: "давай создадим свой скилл".
- **pdf / pptx** - чтение и генерация документов и презентаций.

## Деплой

Платформа деплоя пока не выбрана.

До выбора платформы:
- не деплоить без явной команды;
- перед любым push выполнять `npm run build`;
- если позже появится автодеплой на push в `main`, зафиксировать здесь платформу, команду проверки статуса и порядок чтения логов.
