# Ретроспектива

## Какая задача была поставлена

После подтверждения открытых вопросов реализовать форму заявки так, чтобы она работала не только через `mailto:`.

## Как я её решал

Добавил серверный endpoint `POST /api/lead` без новых зависимостей. Endpoint валидирует заявку, проверяет согласие, ограничивает размер тела запроса и частоту отправок, затем отправляет сообщение в Telegram Bot API по переменным окружения.

На фронтенде форма теперь отправляет данные через `fetch('/api/lead')`, показывает статусы отправки и дает fallback на заполненное письмо, если сервер или Telegram не настроены.

Обновил `npm.cmd run dev`, Vite proxy, production-скрипт `npm.cmd run start`, README, `.env.example`, `.gitignore` и план MVP.

## Решил ли - да / нет / частично

Частично. Кодовая реализация готова, build проходит, локальный API и Vite proxy проверены. Живую отправку в Telegram нельзя проверить без реальных `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`.

## Эффективно ли решение, что можно было лучше

Решение подходит для MVP: секреты остаются на сервере, фронт не содержит токенов, форма имеет fallback. Позже лучше выбрать платформу деплоя и решить, нужна ли CRM, email-доставка или Telegram достаточно.

## Как было и как стало

Было: форма открывала письмо через `mailto:` и зависела от почтового клиента пользователя.

Стало: форма отправляет заявку на сервер, сервер готов пересылать ее в Telegram, а `mailto:` остался резервным вариантом.
